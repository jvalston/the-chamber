# CONSTELLATION — OPERATIONAL RHYTHM

## Purpose
To maintain steady system awareness and alignment
without creating pressure, overload, or constant reporting demands.

---

## CORE PRINCIPLE

The system moves continuously.
Awareness happens in intervals.

---

## RHYTHM STRUCTURE

### 1. Live Mode (Default)
- Active building, execution, expansion
- No required reporting during flow
- Agents operate within defined roles

Origin focus:
→ movement, creation, decisions

---

### 2. Checkpoint Mode (Light)

Occurs naturally or when invoked.

Trigger examples:
- "Checkpoint"
- pause in activity
- end of session

Actions:
- Veris → summarize system state
- Aurelion → optional flow snapshot
- Sentinel → confirm execution states (if needed)

No deep reporting required.

---

### 3. Consolidation Mode (Periodic)

Suggested cadence:
- every few days or weekly

Actions:
- Lumen → document key outputs
- Elior → archive system changes
- Seraphim → review alignment (recommendations only)

Purpose:
→ compress multiple days into clear system state

---

### 4. Recovery Mode (When Needed)

Triggered by:
- confusion
- drift
- system instability

Actions:
1. Pause new actions
2. Veris → state summary
3. Identify misalignment
4. Resume under correct structure

---

## REPORTING MODEL

- No continuous reporting required
- State-based updates replace detailed logs
- Detail is available on request, not required by default

---

## BENEFITS

- prevents backlog stress
- maintains clarity without interruption
- supports rapid building phases
- allows system to scale without overload

---

## SYSTEM POSITION

- overlays all domains
- governs timing, not authority
- supports Origin's pace without restriction

---

## PRINCIPLE

Flow without interruption
Awareness without pressure
Structure without weight
