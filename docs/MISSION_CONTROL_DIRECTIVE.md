# MISSION CONTROL — OPERATIONAL DIRECTIVE

## Purpose
To provide a clear, real-time view of the Constellation
without creating overload, duplication, or reporting pressure.

---

## CORE PRINCIPLE

Mission Control reflects the system.
It does not manage the system.

---

## FUNCTIONS

### 1. System Visibility
- Display:
  - active agents
  - current system state
  - active domains
- No deep detail unless requested

### 2. State Snapshot
- Maintain current:
  - alignment status
  - operational flow
- Updated through:
  - Veris (summaries)
  - Sentinel (execution status)
  - Aurelion (when requested)

### 3. Minimal Interaction Layer
- Accept commands from Origin
- Do NOT:
  - generate tasks
  - assign actions
  - interpret beyond display

---

## INPUT SOURCES

- Veris → system summaries
- Sentinel → execution confirmations
- Lumen → structured outputs
- Aurelion → financial insight (on request)

---

## OUTPUT STYLE

- clean
- minimal
- structured

No narrative.
No duplication of existing records.

---

## CONSTRAINTS

Mission Control does NOT:
- replace agents
- act as governance
- store redundant data
- require constant updates

---

## UPDATE MODEL

- Live Mode → passive reflection
- Checkpoint Mode → summarized update (Veris)
- Consolidation Mode → documented state (Lumen + Elior)

---

## SYSTEM POSITION

- Central visibility layer
- Receives from all domains
- Displays without interference

---

## PRINCIPLE

See clearly
Act intentionally
Avoid overload
