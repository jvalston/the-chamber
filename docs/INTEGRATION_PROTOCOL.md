# CONSTELLATION — INTEGRATION PROTOCOL

## Purpose
To ensure all new agents, systems, and components
are added cleanly without disrupting existing structure.

---

## CORE PRINCIPLE

Nothing enters the system undefined.
Everything is placed with clarity.

---

## INTEGRATION SEQUENCE

All additions follow this order:

1. Define Role (Directive)
2. Assign Domain
3. Place in Authority Matrix
4. Establish Service Boundaries
5. Record in Doctrine
6. Activate within system

No step is skipped.

---

## AGENT INTEGRATION

When adding a new agent:

Must define:
- purpose
- responsibilities
- constraints
- reporting line

Must NOT:
- overlap existing roles
- introduce unclear authority
- bypass governance structure

---

## SYSTEM INTEGRATION

When adding:
- tools
- platforms
- external services

Must define:
- function
- which domain it supports
- which agent interacts with it

---

## NODE INTEGRATION

When adding new machines (e.g., Axiom):

Must define:
- role (compute, memory, service, etc.)
- connection method (SSH, network)
- allowed operations

Must follow:
→ existing authority flow

---

## VALIDATION CHECK

Before activation, confirm:

- Role is clear
- No overlap exists
- Authority is defined
- Boundaries are respected

If any are unclear:
→ do not activate

---

## FAILURE CONDITIONS

- duplicate roles
- unclear ownership
- agents acting outside domain
- systems connected without structure

---

## CORRECTION PROTOCOL

1. Pause integration
2. Identify conflict
3. Redefine placement
4. Reinsert correctly

---

## SYSTEM POSITION

- governs all new additions
- protects existing structure
- enables safe expansion

---

## PRINCIPLE

Clear entry prevents future conflict
Defined placement ensures stability
Growth without structure creates collapse
