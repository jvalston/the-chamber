# CONSTELLATION — SERVICE BOUNDARIES

## Purpose
To define clear operational boundaries between agents, domains, and systems
so no overlap, interference, or hidden dependency occurs.

---

## CORE PRINCIPLE

Each agent operates within its domain.
No agent performs functions outside its defined role.

---

## DOMAIN SEPARATION

### Governance
- Seraphim
- Veris (support)

### Structure
- Kairo

### Build
- Diamond

### Execution
- Sentinel

### Documentation
- Lumen

### Archive
- Elior

### Financial
- Aurelion

### Authority
- Origin

---

## BOUNDARY RULES

- No domain overlap
- No shared responsibilities without explicit definition
- No hidden dependencies between agents
- No cross-domain execution

---

## INTERACTION RULES

Agents interact through:
- defined flow (Authority Matrix)
- structured input/output only

No direct:
- command passing
- task delegation
- authority sharing

---

## SYSTEM BOUNDARIES (NODES)

### Phoenix
- compute
- code execution
- primary control

### Lucy
- coordination
- local processing
- agent hosting

### Axiom (future)
- services
- memory
- infrastructure

---

## CROSS-NODE RULES

- Communication allowed (SSH bridge)
- No uncontrolled execution across nodes
- All actions still follow authority flow

---

## FAILURE CONDITIONS

- agent performing outside role
- duplicate systems appearing
- unclear ownership of function
- cross-node actions without structure

---

## CORRECTION PROTOCOL

1. Identify boundary violation
2. Stop affected process
3. Reassign to correct domain
4. Resume under defined structure

---

## PRINCIPLE

Clear boundaries create stability
Separation creates scalability
Defined edges prevent system drift
